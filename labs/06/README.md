## Задания

### Задание 1: Сравнение подходов к параллельному построению гистограммы изображения — **100 баллов**

#### Цель работы

Исследовать влияние разных способов синхронизации и организации данных на производительность
параллельной программы на примере построения гистограммы RGB-изображения.

#### Постановка задачи

Необходимо написать программу, которая загружает растровое RGB-изображение
большого размера и строит для каждого цветового канала отдельную гистограмму.

Для каждого канала требуется получить массив из 256 значений типа `float` от `0` до `1`,
где каждое значение задаёт относительную частоту появления соответствующей интенсивности в данном канале.

То есть программа должна вычислить:

- `histR[0..255]`
- `histG[0..255]`
- `histB[0..255]`

где сумма значений в каждой гистограмме должна быть равна 1.

#### Что требуется реализовать

Нужно реализовать несколько вариантов, а потом сравнить производительность.

##### Вариант 1. Однопоточная реализация

Базовый вариант для сравнения.

Требования:

- один поток
- обычные неатомарные счётчики
- после подсчёта выполнить нормализацию к диапазону `[0; 1]`

##### Вариант 2. Параллельная реализация с общими атомарными счётчиками

Изображение разбивается между несколькими потоками,
которые запускаются вручную через `std::jthread`.

Все потоки обновляют общую гистограмму с помощью атомарных операций (выберите необходимый memory order).

Нужно реализовать и сравнить **две схемы размещения счётчиков в памяти**.

###### Схема 2a

Общий массив длины `256 * 3`, например:

```cpp
hist[256 * 3]
```

где данные каналов расположены подряд по значениям интенсивности, например:

```cpp
R0, G0, B0, R1, G1, B1, ...
```

###### Схема 2b

Общий массив длины `3 * 256`, например:

```cpp
R[256], G[256], B[256]
```

то есть данные трёх каналов лежат тремя непрерывными блоками.

Требуется сравнить производительность этих двух схем и объяснить различия.

##### Вариант 3. Параллельная реализация с локальными частичными гистограммами

Каждый поток должен:

- обработать свою часть изображения
- построить собственную локальную гистограмму без атомарных операций

После завершения работы потоков локальные гистограммы нужно объединить в одну итоговую.

Требуется:

- реализовать объединение результатов
- сравнить производительность с вариантом 2
- объяснить, за счёт чего возникает разница

#### Требования к реализации

##### Загрузка изображения

Можно использовать один из вариантов:

- готовую библиотеку загрузки изображений
- собственный загрузчик простого формата
- формат BMP как наиболее простой для самостоятельной реализации

Требования:

- изображение должно быть RGB
- размер изображения должен быть достаточно большим, чтобы время вычисления было измеримым

Желательно использовать изображения хотя бы порядка:

- 4000×3000
- 6000×4000
- 8000×8000

Допускается использование синтетически сгенерированных изображений для
проверки краевых случаев (например, когда все пиксели имеют один цвет).

---

##### Формат данных

Рекомендуется использовать целочисленные 32-разрядные счётчики на этапе подсчёта и только потом переводить их в `float`.

---

##### Параллелизм

Потоки должны запускаться вручную.

Нужно самостоятельно:

- выбрать число потоков
- разбить работу между потоками
- дождаться завершения всех потоков

Желательно исследовать несколько значений числа потоков, например:

- 1
- 2
- 4
- 8
- 16

если это имеет смысл на используемой машине.

##### Бенчмарки

Работу необходимо оформить в виде бенчмарков.

Можно использовать:

- Catch2 Benchmark
- Google Benchmark

Нужно измерять отдельно:

- однопоточный вариант
- атомарный вариант со схемой 2a
- атомарный вариант со схемой 2b
- вариант с локальными частичными гистограммами

Необходимо исключить из измерений:

- чтение файла с диска
- выделение памяти
- генерацию тестовых данных

То есть изображение нужно загрузить один раз заранее,
а в бенчмарке измерять только построение гистограммы.

#### Предмет исследования

Необходимо не просто привести числа, а проанализировать результаты.

Обязательно сравнить

1. Однопоточный и параллельные варианты. Насколько параллельные решения ускоряют или замедляют программу.
2. Две атомарные схемы размещения данных. Почему одно размещение может работать хуже другого.
3. Атомарный подход и локальные гистограммы. Какой из вариантов быстрее и почему.
4. Зависимость от числа потоков. Как меняется производительность при увеличении числа потоков.

#### Содержимое отчёта

##### 1. Проверка корректности

Нужно показать, что все варианты дают одинаковый результат с допустимой погрешностью.

Например:

- сравнить суммы гистограмм
- сравнить элементы с эталонным решением
- проверить, что сумма значений каждого канала близка к 1

##### 2. Результаты измерений

Представить таблицы и графики.

Желательно построить:

- время выполнения vs число потоков
- ускорение vs число потоков
- сравнение разных реализаций на одном графике

##### 3. Анализ результатов

Нужно объяснить:

- Как масштабируется атомарное решение
- как различается время работы решения на atomics при разных вариантах размещения гистограмм в памяти
- почему локальные частичные гистограммы обычно выигрывают
- в каких случаях параллельный вариант может оказаться медленнее однопоточного
- Как меняется масштабируемость в зависимости от размеров изображения
- Как на время работы влияет содержимое изображения (реалистичная фотография vs иллюстрация) при одинаковых размерах. Почему?

##### 4. Выводы

Кратко сформулировать:

- какой подход оказался самым быстрым
- какой — самым простым
- какую роль сыграли атомарные операции
- как на производительность влияет организация памяти

### Задание 2 — TicketOffice — **30 баллов**

Реализуйте класс `TicketOffice`, выполняющий продажу билетов в lock-free стиле.
Метод `SellTickets` выполняет продажу билетов по следующим правилам:

- Нельзя продать больше билетов, чем имеется в наличии.
- Параметр ticketsToBuy должен быть положительным.

Прототип класса TicketOffice дан ниже.

```cpp
class TicketOffice
{
public:
    explicit TicketOffice(int numTickets);

    TicketOffice(const TicketOffice&) = delete;
    TicketOffice& operator=(const TicketOffice&) = delete;

    /**
     * Выполняет продажу билета.
     * Возвращает количество фактически проданных билетов.
     * Если ticketsToBuy <= 0, выбрасывается исключение std::invalid_argument
     */
    int SellTickets(int ticketsToBuy);

    int GetTicketsLeft() const noexcept;

private:
    // Количество билетов должно быть атомарным
    std::atomic<int> m_numTickets;
};
```

Расставьте правильным образом порядок доступа к атомарным переменным.

Для проверки работы класса в многопоточном режиме напишите юнит-тесты.

#### Бонус за сравнение производительности с lock-based версией класса — **10 баллов**

Проведите измерения времени работы программы при количестве потоков, изменяющемся от 1 до 30
и сравните его с версией класса `TicketOfficeWithLock`, использующей мьютекс для синхронизации.
Сделайте выводы.

- `RequestStop` для запроса остановки.
- `GetToken` для получения `StopToken`.

`StopToken` должен предоставлять методы:

- `StopRequested` для проверки того, была ли запрошена остановка.
